## 1. Главный вопрос

Перед созданием любого правила задай себе один вопрос:

**«Что я повторяю Cline снова и снова, потому что он этого не знает?»**

Если ответ есть — это кандидат на правило. Если нет — правило не нужно.

---

## 2. Шаг 1: Разделить на два типа

Сначала определи тип правила, потом думай о содержимом.

> **Это нужно при КАЖДОМ запросе?**
> ├── Да → всегда активное правило (без frontmatter)
> │         Примеры: структура проекта, стек, ограничения, соглашения
> └── Нет → условное правило (с `paths:` frontmatter)
>           Примеры: правила для компонентов, API, тестов, БД

---

## 3. Шаг 2: Определить нужно ли правило вообще

Правило нужно только если выполняется хотя бы одно условие:

| Условие | Пример |
|---------|--------|
| Cline делает ошибку без этой информации | Создаёт компоненты в неправильной директории |
| Ты повторяешь одно и то же в каждой сессии | «Не трогай /legacy/» |
| Есть отклонение от стандартного поведения | Используем Prisma, а не Drizzle |
| Есть жёсткое ограничение с причиной | Нельзя менять schema.prisma без миграции |
| Есть паттерн, который нужно воспроизводить | Все ошибки через `/src/lib/errors.ts` |

**Правило НЕ нужно если:**

- Это уже покрыто установленным скиллом (`next-best-practices`, `vercel-react` и т.д.)
- Инструкция слишком расплывчата («пиши хороший код»)
- Это разовая задача — просто скажи в промпте
- Это уже описано в другом правиле

---

## 4. Шаг 3: Применить принцип «одна тема — один файл»

Каждый файл правил должен отвечать на вопрос: **«Что именно я хочу иметь возможность отключить отдельно?»**

**Практический тест:** если ты хочешь отключить правило при прототипировании — это отдельный файл. Если правило всегда нужно вместе с другим — можно объединить.

```
Плохо:  один большой файл rules.md со всем
Хорошо: coding.md, testing.md, architecture.md — каждый можно отключить
```

---

## 5. Шаг 4: Карта категорий правил

Вот исчерпывающий список категорий. Для каждой — вопрос-триггер, который помогает понять, нужна ли она тебе.

### Всегда активные (без frontmatter)

#### `project-overview.md`
**Триггер:** Cline не знает структуру директорий и угадывает пути к файлам?

Нужен всегда. Это минимальный контекст без которого Cline работает вслепую.

#### `conventions.md` или `coding-standards.md`
**Триггер:** Cline генерирует код в стиле, отличном от принятого в проекте?

Нужен если есть отклонения от дефолтного поведения модели: другой стиль именования, запрет определённых паттернов, обязательные практики.

#### `architecture.md`
**Триггер:** Cline предлагает архитектурные решения, противоречащие уже принятым?

Нужен если есть нетривиальные архитектурные решения с причинами («почему Prisma, а не Drizzle»).

#### `memory-bank.md`
**Триггер:** Используешь Memory Bank?

Нужен всегда при использовании Memory Bank — это «мост» между правилами и памятью.

### Условные (с `paths:` frontmatter)

#### `components.md`
**Триггер:** Cline создаёт компоненты без нужной структуры props, без правильного использования `cn()`, с неправильным размером?
```yaml
paths:
  - "src/components/**"
  - "src/app/**/components/**"
```

#### `api-routes.md`
**Триггер:** Cline пишет API routes без валидации, без проверки аутентификации, с неправильным форматом ответов?
```yaml
paths:
  - "src/app/api/**"
```

#### `database.md`
**Триггер:** Cline пишет Prisma-запросы прямо в компонентах или Server Actions, минуя слой репозиториев?
```yaml
paths:
  - "prisma/**"
  - "src/lib/db/**"
  - "src/repositories/**"
```

#### `testing.md`
**Триггер:** Cline пишет тесты с неправильными именами, мокирует не то, использует не тот фреймворк?
```yaml
paths:
  - "**/*.test.ts"
  - "**/*.spec.ts"
  - "**/__tests__/**"
```

#### `styles.md`
**Триггер:** Есть кастомная дизайн-система с нестандартными токенами Tailwind, которые Cline не знает?
```yaml
paths:
  - "tailwind.config.*"
  - "src/styles/**"
```

#### `docs.md`
**Триггер:** Cline пишет документацию в неправильном стиле или формате?
```yaml
paths:
  - "docs/**"
  - "**/*.md"
  - "**/*.mdx"
```

## 6. Шаг 5: Глобальные vs проектные правила

> **Это применимо ко ВСЕМ моим проектам?**
> ├── Да → глобальное правило (`~/.cline/rules/` или `~/Documents/Cline/Rules/`)
> │         Примеры: личный стиль кода, предпочтения по инструментам
> └── Нет → проектное правило (`.cline/rules/`)
>           Примеры: структура этого проекта, его ограничения

Проектные правила коммитятся в репозиторий и шарятся с командой. Глобальные — только на твоей машине.

---

## 7. Шаг 6: Дерево решений для нового правила

```
Есть идея для правила
│
├── Это уже покрыто скиллом? → Не создавай, дублирование тратит токены
│
├── Это применимо ко всем проектам? → Глобальное правило
│
├── Нужно при каждом запросе?
│   ├── Да → Всегда активное (без frontmatter)
│   │         Добавить в project-overview.md или создать отдельный файл
│   └── Нет → Условное (с paths:)
│             Определить glob-паттерн для нужных файлов
│
├── Это одна тема или несколько?
│   ├── Одна → Один файл
│   └── Несколько → Разбить на отдельные файлы
│
└── Правило конкретное и однозначное?
    ├── Да → Создавай
    └── Нет → Уточни или не создавай
```

---

## 8. Шаг 7: Аудит существующих правил

Правила нужно периодически пересматривать. Устаревшее правило хуже отсутствующего — оно активно вводит Cline в заблуждение.

**Вопросы для аудита каждого файла:**

- Это правило всё ещё актуально? (стек не изменился?)
- Cline нарушает это правило? (если нет — может, оно не нужно)
- Это правило дублирует другое?
- Это правило дублирует установленный скилл?
- Файл длиннее 80 строк? (возможно, нужно разбить или сократить)

## Руководство по написанию правил для Cline

### Как правила попадают в системный промпт

Прежде чем писать правила, важно понять, что с ними происходит технически. Каждый файл правила форматируется так:

```
{относительный_путь_к_файлу}
{содержимое_файла}
```

Все правила объединяются через `\n\n` и вставляются в системный промпт в секцию `USER'S CUSTOM INSTRUCTIONS`.

**Это означает:**

- **Имя файла видно модели** — оно становится частью контекста. `components.md` говорит модели «это правила для компонентов» ещё до прочтения содержимого.
- **YAML frontmatter вырезается** — модель видит только тело правила, не frontmatter.
- **Пустые файлы пропускаются** — пустой файл = нет токенов.
- **Отключённые правила пропускаются** — toggle в UI работает на уровне загрузки, не на уровне промпта.

### Анатомия файла правила

#### Тип 1: Всегда активное правило (без frontmatter)

```markdown
# Заголовок правила

Опциональный контекст: зачем это правило существует.

## Раздел 1
- Конкретная инструкция
- Ещё инструкция с примером: `вот так`
- Ссылка на файл в проекте: см. /src/utils/errors.ts

## Раздел 2
- Инструкция с объяснением причины (не модифицировать /legacy — запланировано к удалению в Q3)
```

#### Тип 2: Условное правило (с YAML frontmatter)

```markdown
---
paths:
  - "src/components/**"
  - "src/app/**/components/**"
---

# Заголовок правила

Содержимое правила...
```

Разделители `---` обязательны. Всё после второго `---` — тело правила. Frontmatter вырезается перед отправкой модели.

### Правила написания содержимого

#### 1. Конкретность вместо расплывчатости

**Плохо** — слишком абстрактно, модель не знает что делать:
```markdown
- Используй хорошие имена переменных
- Пиши чистый код
- Следуй лучшим практикам
```

**Хорошо** — конкретные, однозначные инструкции:
```markdown
- Переменные: camelCase, классы: PascalCase, константы: UPPER_SNAKE_CASE
- Максимальная длина функции: 50 строк. Если длиннее — разбить на подфункции
- Запрещено: `any` в TypeScript. Использовать `unknown` + type guard
```

#### 2. Объяснение причины для неочевидных правил

**Плохо** — модель не понимает почему и может обойти правило:
```markdown
- Не изменять /legacy/
- Не использовать fetch напрямую
```

**Хорошо** — причина помогает модели принимать правильные решения в пограничных случаях:
```markdown
- Не изменять /legacy/ (код запланирован к удалению в Q3, любые изменения — потеря времени)
- Не использовать fetch напрямую — только через /src/lib/api.ts (там централизованная обработка ошибок и auth-токены)
```

#### 3. Ссылки на примеры в кодовой базе

Вместо описания паттерна — указать где он уже реализован:

**Плохо** — длинное описание паттерна занимает токены:
```markdown
- При обработке ошибок создавай типизированный объект с полями code, message и details,
  оборачивай в Result<T, E> тип, возвращай из функции вместо throw...
```

**Хорошо** — одна строка, ноль лишних токенов:
```markdown
- Обработка ошибок: следовать паттерну из /src/lib/errors.ts
- Репозитории: следовать структуре /src/repositories/UserRepository.ts
```

#### 4. Краткость — главный приоритет

Правила потребляют токены при каждом запросе (для всегда активных) или при каждом совпадении (для условных). Длинные правила вытесняют полезный контекст из окна.

**Антипаттерны:**
- Вставлять целые style guide'ы или документацию
- Дублировать то, что уже есть в скиллах (`next-best-practices` и т.д.)
- Писать длинные объяснения вместо коротких инструкций
- Повторять одно и то же в разных файлах

**Целевой размер:** всегда активное правило — до 30–50 строк. Условное — до 50–80 строк.

#### 5. Один файл — одна тема

Разбивай по темам, а не по размеру. Это позволяет:
- Отключать отдельные правила через toggle в UI
- Делать правила условными по нужным путям
- Легко находить и обновлять конкретное правило

#### 6. Актуальность важнее полноты

Устаревшее правило хуже отсутствующего — оно активно вводит модель в заблуждение. Правило нужно удалять или обновлять при:
- Смене технологии (перешли с Jest на Vitest)
- Рефакторинге структуры директорий
- Изменении архитектурных решений
- Удалении ограничений

### Условные правила: синтаксис paths

`paths` — единственный поддерживаемый условный оператор. Правило активируется если хотя бы один паттерн совпадает с хотя бы одним файлом в текущем контексте.

#### Таблица glob-паттернов

| Паттерн | Что совпадает |
|---------|---------------|
| `src/**/*.ts` | Все .ts файлы рекурсивно в src/ |
| `**/*.test.ts` | Тест-файлы в любом месте проекта |
| `*.config.js` | Конфиг-файлы только в корне |
| `src/components/*.tsx` | .tsx файлы прямо в components/ (не вложенные) |
| `packages/{web,api}/**` | Файлы в web или api пакетах |
| `src/app/api/**` | Всё в директории api/ рекурсивно |

#### Что считается «текущим контекстом»

Правило активируется если совпадение найдено в любом из:
- Пути файлов, упомянутых в твоём сообщении («обнови src/App.tsx»)
- Открытых вкладках редактора
- Видимых файлах в активных панелях
- Файлах, которые Cline уже изменил в этой задаче
- Файлах, которые Cline собирается изменить

**Практический вывод:** явно упоминай пути в промпте. «обнови src/services/user.ts» надёжно активирует правило; «обнови сервис пользователей» — может не активировать.

#### Специальные значения

```yaml
# Никогда не активируется — способ временно отключить правило без удаления файла
paths: []

# Нет frontmatter вообще — всегда активно
(файл без ---...--- блока)
```

### Структура markdown в правиле

Cline читает правила как контекст — форматирование влияет на качество следования инструкциям:

```markdown
# Название правила
(Заголовок H1 — тема правила целиком)

Одна строка контекста: зачем это правило (опционально).

## Категория
(Заголовок H2 — группа инструкций)

- Инструкция с примером кода: `const x = fn()`
- Инструкция со ссылкой: см. /src/lib/utils.ts
- Инструкция с причиной (причина в скобках)

## Запреты
(Отдельная секция для того, что нельзя делать)

- НЕ использовать X (потому что Y)
- НЕ изменять /path/to/file (потому что Z)
```

Заголовки помогают модели понять область применения каждой инструкции. Маркированные списки делают отдельные требования чёткими. Примеры кода показывают точно что нужно.

### Диагностика проблем

#### Правило не активируется (условное)
- Проверь что паттерн совпадает с реальным путём файла — `**` рекурсивный, `*` нет
- Убедись что правило включено в Rules panel (toggle)
- Проверь синтаксис YAML: правильные `---` разделители, кавычки вокруг паттернов со спецсимволами

#### Frontmatter отображается в выводе
YAML не распарсился. Cline работает в режиме fail-open — показывает сырое содержимое включая frontmatter. Проверь синтаксис:

```yaml
# Правильно
paths:
  - "src/**"
  - "*.config.ts"

# Неправильно — нет кавычек вокруг {
paths:
  - src/{a,b}/**
```

#### Правило активируется неожиданно
`**` очень широкий паттерн. `**/*.md` совпадёт с любым markdown-файлом в проекте, включая README.md в корне. Сужай паттерны:

```yaml
# Слишком широко — активируется при любом .md файле
paths:
  - "**/*.md"

# Точнее — только документация
paths:
  - "docs/**"
  - "docs/**/*.md"
```

#### Тест нового паттерна

Не уверен совпадает ли паттерн? Создай тестовое правило:

```yaml
---
paths:
  - "твой/паттерн/**"
---
TEST: это правило должно активироваться для файлов в твой/паттерн/
```

Открой файл по этому пути и проверь появится ли уведомление «Conditional rules applied: workspace:test.md».

## Чеклист перед сохранением правила

- [ ] Каждая инструкция конкретна и однозначна
- [ ] Неочевидные ограничения объяснены (причина в скобках)
- [ ] Паттерны в кодовой базе — ссылки на файлы, не описания
- [ ] Нет дублирования с установленными скиллами
- [ ] Нет дублирования с другими правилами
- [ ] Файл не длиннее 50–80 строк
- [ ] Имя файла отражает тему правила
- [ ] Условное правило — паттерны paths протестированы
